## *The Engineer's View: What the Code Actually Shows*

### 2.1 The 269 Checks ‚Äî What They Mean

The CheckName enum containing 269 verification checks is not a surveillance system designed to catch bad guys. In KYC engineering, comprehensive checks are **standard compliance practice** for any platform with financial exposure. Most checks (AAMVA database lookup, SSN comparison, MRZ detection) are entirely standard.

**What's NOT standard:**

`SelfieSuspiciousEntityDetection` ‚Äî This check exists in the selfie verification category. KYC industry norms flag specific criteria: known fraud patterns, synthetic faces, liveness failures. The fact that the criteria are not documented in the code means they're either:
- Defined externally in a configuration layer (normal)
- Defined in a model whose criteria are opaque (concerning)
- Deliberately obfuscated (the article mentions code that "depends on obfuscation for encryption keys")

The inability to audit what makes a selfie a "suspicious entity" is a real accountability problem regardless of intent.

**PEP Facial Recognition** ‚Äî Comparing user selfies to politicians via Wikidata is ostensibly for catching bad actors impersonating public figures or catching sanctioned politicians. But it generates similarity scores, not binary matches. A "High" similarity score to a world leader ‚Äî what happens to that data? Does it influence the verification result? The code doesn't answer this.

### 2.2 The Infrastructure Architecture ‚Äî Why It's Different

Standard KYC vendors run behind shared Cloudflare infrastructure. This keeps costs low and scales efficiently. The decision to run `openai-watchlistdb.withpersona.com` on:
- **Dedicated GCP** (not shared)
- **Envoy/Istio service mesh** (enterprise-grade traffic management)
- **Separate from Cloudflare** (means the data is compartmentalized from Persona's normal operations)

...indicates OpenAI requested ‚Äî or Persona designed ‚Äî intentional separation of this specific data from the rest of the business. This is the kind of architecture decision you make when a legal or security team says "this data cannot co-mingle."

The 18-month gap (November 2023 start vs. April 2025 public announcement) means this system was operational for OpenAI for over a year and a half before users were told they were being verified. **The screening happened before the disclosure.**

### 2.3 SAR/STR Filing ‚Äî The Government Reporting Pipeline

Filing a SAR (Suspicious Activity Report) with FinCEN is a regulated activity. Under the Bank Secrecy Act, financial institutions must file SARs for transactions over $5,000 that may involve illegal activity. The fact that a KYC platform has SAR filing capability doesn't automatically make it sinister ‚Äî it's legally required for certain use cases.

**What raises questions:**

1. **STR filing to FINTRAC with intelligence program codenames** ‚Äî Filing to Canada's financial intelligence unit is standard for companies operating in Canada. But the use of `publicPrivatePartnershipProjectNameCodes` with names like LEGION, SHADOW, GUARDIAN suggests these are not ad-hoc labels but formal inter-agency programs. These codenames represent actual collaboration frameworks between financial institutions and intelligence agencies.

2. **The cross-border capability** ‚Äî A platform that files SARs to FinCEN AND STRs to FINTRAC has cross-border financial intelligence reporting. Combined with the government platform, this means a flagged user potentially generates financial intelligence reports in multiple jurisdictions simultaneously.

3. **No appeal, no notification** ‚Äî In a regulated SAR framework, the subject of a SAR is **not notified** (it's legally prohibited to tip off the subject). If the same framework applies here, denied OpenAI users may have generated financial intelligence reports they cannot know about, cannot appeal, and cannot correct.
